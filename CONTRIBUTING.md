# Contributing to Forge

Thanks for your interest. Forge is a token-efficient developer-tooling suite for
AI coding agents: a single `frg` CLI plus an MCP server that turns noisy build,
test, lint, search, and project-analysis output into compact structured JSON.

We welcome contributions of all sizes — bug reports, documentation, new
wrappers, parser improvements, MCP compatibility fixes, and better diagnostics.

## Code of Conduct

This project follows the [Contributor Covenant 2.1](CODE_OF_CONDUCT.md).
Participation is conditioned on adherence to its terms.

## Getting Started

### Prerequisites

- Rust stable or nightly with `rustfmt` and `clippy`
- A POSIX-ish environment (Linux, macOS)
- Optional: a running Ferrosa Memory instance for ingestion/task-store features

### Building

```bash
git clone https://github.com/ferrosadb/forge.git
cd forge
cargo build --workspace
```

### Running the Test Suite

```bash
cargo test --workspace
cargo test -p forge-test-summary
```

Some commands shell out to external toolchains (`cargo`, `go`, `npm`, `pytest`,
`mix`, etc.) or connect to Ferrosa Memory. Unit tests should not require those
services unless explicitly marked and documented.

### Hygiene Checklist (Before Every Commit)

1. `cargo fmt --all -- --check` — format
2. `cargo clippy --workspace --all-targets -- -D warnings` — lint, zero warnings
3. `cargo test --workspace` — all tests pass
4. `cargo build --workspace --all-targets` — full workspace compiles clean

CI runs the same core gates on every pull request.

## Development Workflow

### Branching

- Fork the repository and create a topic branch off `main`
- Branch naming: `feat/short-description`, `fix/issue-summary`, `docs/what-changed`
- Never commit directly to `main`

### Commit Messages

Conventional-commit style:

- `feat(crate): summary` — new feature
- `fix(crate): summary` — bug fix
- `refactor(crate): summary` — restructure without behavior change
- `test(crate): summary` — test-only change
- `docs: summary` — documentation
- `chore: summary` — tooling, deps, build

Body explains *why*. The diff already shows the *what*.

### Test-Driven Development

For new tools and bug fixes, write the failing test first, then make it pass,
then refactor. For bug fixes specifically: include a regression test.

### Pull Requests

- Keep PRs focused. One bug fix or one feature per PR.
- Include test coverage. PRs without tests for new behavior will be requested
  to add them.
- Link the issue number if one exists: "Closes #123".
- Mark drafts as drafts. Mark ready-for-review when CI is green.

PRs are squash-merged by default.

## Areas Where Help Is Welcome

- **MCP client compatibility** — Claude Code, Claude Desktop, and other MCP hosts
- **Parser quality** — tighter summaries for test runners, linters, and build logs
- **Language wrappers** — Go, Python, Node, .NET, Elixir, and Rust tool output
- **Security tooling** — secret scanning and dependency-audit coverage
- **Documentation** — examples, integration guides, and real workflow recipes

## What We Don't Accept

- Code generated by AI tools without human review and understanding. Using AI
  as an assistant is fine; the contributor must understand and stand behind
  every line they submit.
- Drive-by formatting changes that touch unrelated files.
- Dependency bumps without a stated reason.
- Changes that disable or skip existing tests without replacing them.

## Reporting Bugs

Open a GitHub issue with:

- Forge version (commit hash or release tag)
- Command or MCP tool used
- Reproduction steps, ideally a minimal self-contained example
- Expected vs actual behavior
- Relevant logs (sanitize secrets first)
- Environment: OS, Rust version, and any external tool version involved

Security issues — please do **not** open a public issue. See [SECURITY.md](SECURITY.md).
